# 需求文档

## 简介

当前 Homepage 仪表盘的配置完全依赖 `config/` 目录下五份 YAML 文件（`settings.yaml`、`services.yaml`、`bookmarks.yaml`、`widgets.yaml`、`docker.yaml`）。服务端通过 `loadConfig()` 在每次数据 API 请求时重新读盘、规范化并重建 AllowList；前端仅通过 `GET /api/config` 获取浏览器安全的规范化配置快照，页面无编辑与写回能力。密钥字段在 YAML 中以 `${ENV_VAR}` 整值占位，仅服务端插值，明文永不进入浏览器。

本特性目标是以**尽量简单**的方式重构配置管理：用户在前端通过**结构化友好界面**（表单、分组列表、字段控件，以及新增/删除/调序等操作）编辑配置后实时保存，后端立即持久化并生效；配置编辑器**不是** YAML 源码编辑器，也**不是**粘贴/编辑 YAML 文本的体验，用户**无需了解 YAML 语法**即可完成前端配置修改；无需登录服务器手工改 YAML、无需重启服务。YAML 仅作为配置目录中配置五文件的持久化格式，由服务端读写与解析，不得作为前端的主要编辑面。持久化仍落在现有配置目录与五文件模型上，复用现有规范化、AllowList 与安全校验管线；不引入独立数据库，不改变代理门禁与密钥不进浏览器的安全边界。

配置编辑器 UI 必须遵循当前仪表盘的技术栈与视觉/交互约定（React、Tailwind CSS、CSS 主题令牌、shadcn/ui），与主界面保持一致，不得另起一套 UI 体系、组件库或平行设计语言；也不得重新引入旧版 Homepage UI、旧 CSS，或照搬参考实例的视觉风格。

第一阶段范围包括：面向非技术用户的、与仪表盘同一技术栈实现的结构化可编辑配置界面、校验后的实时保存写回、保存成功后后端立即按新配置鉴权与提供数据、前端在保存成功后刷新展示、密钥字段的安全编辑策略，以及失败时的中文错误反馈与配置不损坏保证。

第一阶段非目标：多用户认证与权限、配置版本历史/回滚 UI、多人协同编辑冲突解决、文件监听热推送（WebSocket）、将密钥明文下发给浏览器、扩大服务组件类型或恢复 ICMP/Docker stats 等既有推迟能力；另建独立设计系统、第二套组件库，或与当前仪表盘视觉/交互约定不一致的配置编辑 UI。

## 术语

| 术语 | 说明 |
| --- | --- |
| 仪表盘 | 现有单页应用，展示服务、书签、信息组件等 |
| 配置目录 | 由 `CONFIG_DIR` 或默认 `config` 解析出的服务端配置根目录 |
| 配置五文件 | `settings.yaml`、`services.yaml`、`bookmarks.yaml`、`widgets.yaml`、`docker.yaml` |
| 配置加载器 | 读取、解析、校验并规范化配置，输出规范化配置与当次 AllowList 的服务端模块 |
| 规范化配置 | 供前端展示与部分 API 使用的浏览器安全配置视图（`NormalizedConfig`），不含密钥明文 |
| AllowList | 仅服务端持有的当次允许列表，含探测目标、组件目标、Docker 端点与已解析密钥 |
| 可编辑配置 | 前端表单绑定与写回 API 使用的结构化配置表示（对象/列表/字段），不是 YAML 源码文本；服务端可将其序列化为配置五文件并再次被配置加载器加载 |
| 配置编辑器 | 仪表盘内以表单、分组列表、输入框、选择器、开关及新增/删除/调序控件呈现的结构化编辑界面；用于查看与修改可编辑配置；不是 YAML/文本代码编辑器，用户无需理解 YAML 语法；与仪表盘共用同一设计体系（React、Tailwind CSS、CSS 主题令牌、现有 shadcn/ui 原语及主题能力），不得另起 UI 体系 |
| 实时保存 | 用户在配置编辑器中提交保存后，客户端立即调用写回接口，无需额外“发布”步骤 |
| 立即生效 | 写回成功后，后续数据 API 请求使用新持久化配置重建 AllowList 并返回新结果，无需重启进程 |
| 密钥字段 | 密码、API key、token、username 等仅服务端持有的敏感字段；存储侧可继续使用 `${ENV_VAR}` 占位 |
| 密钥占位 | 整值形式为 `${ENV_VAR}` 的密钥引用，由服务端用环境变量插值 |
| 密钥掩码 | 前端展示密钥字段时使用的非明文占位提示，不暴露真实密钥值 |
| 写回接口 | 接收可编辑配置、校验并持久化到配置五文件的服务端 API |
| 原子写回 | 先写入临时文件再替换目标文件，避免写到一半导致文件损坏 |
| 配置快照 | 某次 `GET /api/config` 返回的规范化配置 |

## 需求

### 需求 1：配置存储模型与简单重构边界

**用户故事：** 作为维护者，我希望在不大改架构的前提下支持前端改配置并后端生效，以便用最低成本去掉“只能手改 YAML”的负担。

#### 验收标准

1. 系统必须继续以配置目录（由 `CONFIG_DIR` 或默认 `config` 解析）下的配置五文件（`settings.yaml`、`services.yaml`、`bookmarks.yaml`、`widgets.yaml`、`docker.yaml`）作为唯一持久化真相来源；不得引入数据库、独立键值存储或其他非上述五文件的持久化介质，作为第一阶段的配置真相源。
2. 系统必须通过现有配置加载器完成配置的读盘、YAML 解析、受支持字段子集规范化、`${ENV_VAR}` 整值插值、安全视图校验与当次 AllowList 重建；写回成功后的配置读取与数据 API 鉴权路径必须调用与现有 `loadConfig()` 相同的加载语义（包含上述步骤），且不得另建绕过该加载器的并行读取路径或并行鉴权真相源。
3. 当写回接口已成功将可编辑配置持久化到配置五文件后，系统必须使随后到达的每一条数据 API 请求（至少包括：配置读取、HTTP 探测、Docker 状态、信息组件、服务组件）在处理该请求时从当前磁盘重新执行 `loadConfig()`（或等价调用同一配置加载器完整管线），并按本次加载得到的规范化配置与 AllowList 进行鉴权与返回；上述立即生效不得以重启 Node 进程为前提；系统不得将某次请求产生的 AllowList 或规范化配置跨请求复用为后续请求的鉴权或配置真相。
4. 系统不得在第一阶段实现以下能力：基于 WebSocket 或 SSE 的配置变更推送、多用户认证与权限控制、配置版本历史或版本库，以及将 AllowList 或服务端已解析的密钥明文下发至浏览器。
5. 浏览器收到的入口页面 HTML、配置快照、可编辑配置、写回成功或失败响应、错误信息，以及任意 API 响应正文与响应头中，不得包含服务端经配置加载器插值后得到的完整明文敏感值（包括密码、令牌、API 密钥，以及当次 AllowList 内已解析密钥与敏感请求头值）；对外仅允许出现未解析的密钥占位、密钥掩码或“已配置/未配置”状态，不得出现可用于还原完整明文的完整密钥值。
6. 若写回成功后的某次后续数据 API 在重新执行 `loadConfig()` 时失败，则系统必须使该请求返回可观察的失败结果（不得伪造成功数据），且不得回退使用写回前缓存于进程内的 AllowList 或规范化配置作为该请求的鉴权或配置真相；其他并发请求仍必须各自基于其处理时点的磁盘配置五文件经 `loadConfig()` 的结果工作。

### 需求 2：获取可编辑配置

**用户故事：** 作为用户，我希望在前端打开与当前后端一致的可编辑配置，以便在结构化表单界面上查看并修改仪表盘内容，而无需阅读或编辑 YAML。

#### 验收标准

1. 当用户打开配置编辑器时，系统必须基于当前配置目录中的配置五文件提供可编辑配置；在现有配置加载器已支持的字段子集上，该可编辑配置的内容必须与对同一磁盘内容解析并规范化后的结果一致；任一配置五文件缺失时，系统必须按现有配置加载器对该文件的空值或默认处理生成对应部分；且不得要求重启服务进程。
2. 可编辑配置必须覆盖第一阶段与现有配置加载器相同的受支持字段子集，类别必须包括：设置（站点标题、背景与布局相关设置）、服务分组与服务项、书签分组与书签项、信息组件、以及 Docker 端点声明；系统必须将上述内容作为结构化数据（对象、列表与字段）提供给配置编辑器做表单绑定，不得以 YAML 源码文本作为浏览器侧的主要编辑载荷；系统不得要求浏览器直接访问配置目录或配置五文件来获得上述内容；可编辑配置与配置快照必须可区分，且二者均不得承担下发密钥明文的职责。
3. 当某密钥字段在存储侧存在非空值（含密钥占位或字面量密钥）时，返回给浏览器的可编辑表示必须使用密钥掩码或「已配置」状态；当该字段在存储侧缺失或为空时，必须表示为「未配置」或等价的空可编辑状态；系统不得返回环境变量插值后的明文，不得返回与存储值全文相同、可用于还原明文或占位全文的完整密钥值，也不得返回 AllowList 中的已解析密钥。
4. 若获取可编辑配置时，配置五文件存在 YAML 语法错误、顶层结构错误，或出现导致现有配置加载器失败的错误，则系统必须返回结构化简体中文错误（至少包含出错文件名与原因；在可定位时还必须包含字段路径或行列位置），且错误内容中不得包含完整明文密钥；配置编辑器必须展示该错误并给出可操作的下一步提示（例如提示维护者在服务端检查所指配置文件的结构或语法后重试；该提示面向磁盘侧维护场景，不得引导用户在浏览器内编辑 YAML），不得整页白屏，也不得呈现为已成功加载可编辑配置。
5. 当获取可编辑配置成功时，系统必须返回可供配置编辑器做表单与列表控件绑定的结构化数据（非 YAML 文本）；在用户未修改任何字段、且密钥字段均标记为保持原值的前提下，将该数据经合法保存写回后，配置加载器必须能成功加载；写回后受支持字段子集的语义必须与保存前一致，且磁盘上各密钥字段的存储值必须与保存前完全相同。
6. 当配置五文件全部缺失或内容均为空、且现有配置加载器仍按空值或默认处理并成功完成加载时，系统必须返回成功的可编辑配置（各对应部分为空值或默认），配置编辑器必须可打开并以结构化表单/列表界面允许用户从该空白或默认状态开始编辑，不得要求用户粘贴或编写 YAML。

### 需求 3：前端配置编辑体验

**用户故事：** 作为用户，我希望在页面上通过友好的表单与列表直接增删改服务、书签、信息组件与基础设置，以便无需了解 YAML、也无需登录服务器改配置文件。

#### 验收标准

1. 当用户从仪表盘主视图触发配置编辑入口时，系统必须打开配置编辑器；配置编辑器必须提供可触发的退出入口；当用户触发退出时，系统必须关闭配置编辑器并回到仪表盘主视图。
2. 配置编辑器必须是结构化友好界面：对第一阶段全部可编辑类别（设置、服务、书签、信息组件、Docker 端点）使用带中文标签的分组区块、表单字段与列表控件呈现；字段级编辑必须使用输入框、选择器、开关等可视化控件，而不是让用户编写自由格式配置文本；系统不得将原始 YAML 文本域、代码编辑器或“粘贴 YAML”作为主要编辑面，也不得要求用户理解或手写 YAML 语法才能完成编辑。
3. 配置编辑器必须支持通过上述结构化控件编辑与第一阶段配置加载器受支持字段子集一致的下列内容：站点标题、背景、是否等高布局，以及布局中各服务分组的最大列数；服务分组名称，以及组内服务项的名称、链接、打开方式、图标、描述、HTTP 探测相关字段、Docker 引用（端点名与容器名）、受支持服务组件类型声明；书签分组名称，以及组内书签项的名称、链接、打开方式、图标、缩写、描述；信息组件的类型及其受支持选项；Docker 端点的名称与连接串。
4. 配置编辑器必须支持对服务分组、服务项、书签分组、书签项、信息组件通过可见的新增、删除与调序控件进行操作，并对各项字段通过表单控件进行修改；非技术用户必须能仅凭界面标签与控件完成上述操作而无需编辑 YAML；当保存成功后，持久化结果必须反映上述变更，且分组之间及同一分组内条目的相对顺序必须与用户触发保存时编辑器中的顺序一致。
5. 当用户编辑密钥字段时，配置编辑器必须默认以密钥掩码或空输入展示，且不得预填或回显明文密钥；仅当用户主动输入新的非空值时，本次保存载荷才将该新值作为该字段的提交值；对用户未修改的密钥字段，保存载荷必须标记为“保持原值”，且系统不得要求用户重新输入已有密钥才能保存。
6. 当用户触发保存时，配置编辑器必须先执行客户端校验；若存在去除首尾空白后名称为空、作为链接的字段不是绝对 HTTP/HTTPS URL、或其他第一阶段必填项缺失等未通过项，则系统必须在对应字段或表单区域展示简体中文提示，且不得发起写回请求。
7. 配置编辑器内面向用户的按钮、标签、分组标题、校验错误、加载失败与列表空状态文案必须使用简体中文；面向最终用户的提示不得以“请编辑 YAML”“请粘贴配置源码”等要求用户处理 YAML 源码的方式表述。
8. 配置编辑器必须基于当前项目既有前端技术栈实现：React SPA、Vite、Tailwind CSS、CSS 主题令牌，以及项目内已有的 shadcn/ui 原语；必须与仪表盘主界面共用同一套视觉与交互约定，不得引入第二套组件库、平行设计语言或独立主题体系。
9. 配置编辑器中的通用交互控件（按钮、对话框/抽屉、输入框、下拉菜单、加载骨架、提示、分隔线、命令面板等）必须优先使用项目内 `apps/web/src/components/ui` 下已有的 shadcn/ui 原语；不得为上述通用交互另写一套与现有原语并行的基础组件实现。
10. 配置编辑器中业务专属的布局、分组区块与列表编排必须使用 Tailwind CSS 与现有 CSS 主题令牌，并与仪表盘主界面（如页眉、分区、卡片间距与排版节奏）保持一致；仅当 shadcn/ui 原语不适合该业务呈现时才以 Tailwind + 主题令牌组合实现，不得为迁就组件库而强行套用不匹配的卡片/徽章等模式。
11. 配置编辑器必须通过现有主题提供者与 CSS 主题变量支持浅色/深色主题，界面颜色、边框、背景与文字对比必须随主题令牌变化；不得使用会破坏深色模式可读性或与主界面主题切换脱节的硬编码颜色作为主要配色来源。
12. 配置编辑器不得重新引入旧版 Homepage 的 UI 组件或旧 CSS，不得照搬参考实例的视觉风格作为本特性的界面实现依据；界面实现必须与当前仪表盘代码库中的约定一致。

### 需求 4：实时保存与后端持久化

**用户故事：** 作为用户，我希望在结构化编辑界面中点击保存后配置立即写入后端并生效，以便改完就能在仪表盘上看到结果，而无需接触 YAML 文件。

#### 验收标准

1. 当用户在配置编辑器中触发保存且客户端校验全部通过时，系统必须在无中间确认、无额外手工步骤的前提下，向写回接口提交当前完整的结构化可编辑配置（表单/列表绑定的对象数据，而非用户手写的 YAML 文本）；不得要求用户导出文件、手动上传 YAML/配置文件或重启服务。
2. 写回接口必须在修改任一配置五文件之前，对请求中的可编辑配置执行服务端校验；仅当校验全部通过后才允许开始持久化。校验必须覆盖：结构合法性（服务端可序列化为配置五文件并具备合法顶层结构）、第一阶段受支持字段及其值约束，以及“按写回结果走与现有配置加载器等价的规范化路径能够成功完成”。
3. 当服务端校验通过时，写回接口必须由服务端将结构化可编辑配置序列化为 UTF-8 编码的 YAML，并写入配置目录下对应的配置五文件（`settings.yaml`、`services.yaml`、`bookmarks.yaml`、`widgets.yaml`、`docker.yaml`）；YAML 序列化与落盘仅发生在服务端持久化路径，不得作为前端编辑面；每个文件的内容必须能被现有配置加载器按该文件名正确识别并加载。
4. 写回接口对每个目标配置文件必须使用原子写回（先写入同配置目录内的临时文件，再替换目标文件）。若写回过程中发生失败，每个配置文件必须要么保持写回前的完整内容，要么成为本次写回后的完整新内容，不得留下截断、部分写入或语法残缺的 YAML。
5. 当用户保存时对某密钥字段标记为“保持原值”时，写回接口必须在服务端读取该字段写回前的磁盘存储值，并在持久化结果中原样保留该值（原密钥占位或原字面量均按写回前磁盘内容）；不得清空、不得省略以致丢失、不得将密钥掩码或“已配置/未配置”等展示用字符串写入磁盘。不得依赖客户端回传明文或掩码来还原原值。
6. 当用户为某密钥字段提交新的非空值（非“保持原值”）时，系统必须将该新值写入对应配置文件中的存储表示；若该新值整值匹配密钥占位形式，则必须按占位字符串原样存储，并继续仅在服务端做插值。写回接口的成功响应与失败响应均不得回显该新值明文，也不得回显写回前的明文密钥。
7. 如果服务端校验失败，则写回接口必须返回结构化简体中文错误（至少包含失败原因；在能够定位时还必须包含字段路径或所属配置类别/对应配置文件名），必须不修改任一配置五文件的现有内容，且配置编辑器必须展示该错误并保留用户当前全部编辑内容以供继续修改后再次保存。
8. 如果配置目录不存在或不可用、目标路径不可写、临时文件创建失败或原子替换失败，则写回接口必须返回简体中文错误并表明保存未成功，不得返回成功状态；同时必须保证任一已触及的配置文件仍满足“完整旧内容或完整新内容”的约束，不得留下截断或半写入的 YAML。
9. 当且仅当服务端校验通过且所有本次写回的目标配置五文件均已成功完成原子写回时，写回接口必须返回成功状态；配置编辑器必须在收到该成功响应后向用户展示简体中文成功反馈。若任一时目标文件写回失败，则本次保存必须视为失败并返回中文错误，不得声称保存成功。
10. 如果用户对某密钥字段提交的新值为空字符串（非“保持原值”），则写回接口必须将该字段按空值持久化（写入空字符串，或省略该键且与配置加载器对缺省/空密钥的既有语义一致），不得继续保留写回前的旧密钥存储值。
11. 写回接口在同一次保存请求中必须基于同一份已通过服务端校验的可编辑配置生成并写回全部目标配置文件内容；不得在仅部分类别文件已更新的情况下返回成功状态。

### 需求 5：保存后立即生效与界面同步

**用户故事：** 作为用户，我希望保存成功后仪表盘马上按新配置工作，以便立刻验证改动。

#### 验收标准

1. 当写回接口返回成功后，系统必须保证任意在该成功响应产生之后开始处理的配置读取与数据 API 请求（含 HTTP 探测、Docker 状态、信息组件、服务组件）均通过 `loadConfig()` 基于磁盘上已更新的配置五文件重建规范化配置与 AllowList，并据此鉴权与返回结果，且不得要求重启 Node 进程。
2. 当写回接口返回成功后，若新配置使某探测目标、组件目标或 Docker 引用相对写回前发生新增、变更或移除，则在该成功响应之后开始处理的对应数据 API 请求必须按新 AllowList 允许或拒绝；已从配置中移除且不再出现于新 AllowList 的目标必须被拒绝，不得再被代理访问。
3. 当写回接口返回成功后，配置编辑器或仪表盘必须在不要求用户手动整页刷新浏览器的前提下，重新获取配置快照，或使用与该配置快照在受支持字段子集上语义等价的保存结果更新前端状态，使主界面展示的服务、书签、信息组件与设置与刚保存的受支持字段子集一致。
4. 当写回接口返回成功后，若新规范化配置中的站点标题在去除首尾空白后非空，则仪表盘必须将浏览器文档标题更新为该去空白后的值；若未提供有效标题（缺失、空串或仅空白），则仪表盘必须将浏览器文档标题设为既有非空中文默认标题。
5. 在写回接口尚未返回成功之前，数据 API 必须继续仅以当前磁盘上的配置五文件经 `loadConfig()` 的结果作为鉴权与数据真相；当写回校验失败、写回失败或任一目标文件未能完成原子替换时，系统不得在进程内保留或使用与磁盘配置五文件不一致的配置或 AllowList 作为后续请求的鉴权真相。

### 需求 6：并发保存与一致性

**用户故事：** 作为部署者，我希望在偶尔连续保存或重叠请求时配置文件仍保持一致，以便避免损坏或静默丢更新。

#### 验收标准

1. 当两个或更多写回请求在处理时间上重叠（任一写回仍在服务端处理期间又收到另一写回）时，系统必须保证配置五文件作为整体的最终持久化结果，完整等于其中某一次已通过服务端校验并成功落盘的写回载荷所对应的五文件内容；不得出现部分文件来自写回 A、其余文件来自写回 B 的交叉混写；在任意可观察时刻，每个配置文件必须是某次完整写回前的内容或某次完整成功写回后的新内容，不得处于截断或半写入状态。
2. 在配置编辑器对上一次保存请求尚未收到完成响应（成功或失败）期间，系统必须禁用保存操作入口，忽略用户在此期间的重复保存激活，且不得并行发起第二次写回请求；当该次保存请求完成后，系统必须重新启用保存操作入口，并向用户展示与该次结果对应的中文成功或失败反馈。
3. 当写回接口对某次请求返回成功且在此之后尚未发生另一次成功写回时，系统必须使紧接着的一次配置加载（`loadConfig()` 或等价的读盘、解析与规范化）得到与该次写入的受支持字段子集语义一致的规范化结果；上述一致性的前提是环境变量与未被该次写回改动的密钥占位保持不变。
4. 若服务端在已有写回尚未结束时收到另一写回请求，则系统必须将该请求串行化处理或拒绝该请求：串行化时每次写回必须对配置五文件整体依次完整成功或完整失败且互不交错写入；拒绝时必须返回简体中文错误、不得声称保存成功，且不得修改任一配置五文件。
5. 系统不得在第一阶段提供多写回之间的字段级合并或多人协同冲突解决；后完成的成功写回可以整体覆盖先完成的成功写回，但必须仍满足本需求关于五文件整体一致、无交叉混写与无截断文件的约束。

### 需求 7：安全与密钥编辑

**用户故事：** 作为部署者，我希望前端改配置不会把密钥泄露到浏览器，以便继续安全地在局域网使用仪表盘。

#### 验收标准

1. 当系统返回可编辑配置、配置快照或写回成功/失败响应时，系统必须在响应正文与响应头中均不得下发服务端已解析或自磁盘读出的完整明文密钥；对 password、key、apiKey、token、username 等密钥字段，若存储侧为非空，则浏览器侧表示必须仅为密钥掩码或“已配置”状态，若存储侧为空或未设置，则必须为“未配置”状态或空输入表示，且不得包含可用于还原完整密钥的值。
2. 系统必须允许密钥字段在配置五文件中继续以整值密钥占位形式持久化；配置加载器必须仍仅在服务端对占位做环境变量插值；返回给浏览器的可编辑配置、配置快照与写回响应不得包含插值后的明文，前端展示也不得将占位显示为解析后的明文。
3. 当用户在密钥字段提交新的非空字面量且写回成功时，系统必须将该字面量写入对应配置文件的存储表示；该完整字面量不得出现在本次写回响应，以及随后任意一次可编辑配置或配置快照响应中，对应字段必须以密钥掩码或“已配置”状态表示。
4. 若写回请求中包含 AllowList、已解析密钥表或任何试图绕过配置加载器规范化的旁路字段，则系统必须忽略这些字段，不得将其写入配置五文件，也不得将其用于构建当次 AllowList 或数据 API 鉴权；写回成功后的 AllowList 必须仅根据持久化后的配置五文件经配置加载器重建。
5. 系统必须保持既有代理门禁：对未在当次 AllowList 登记的 HTTP 探测目标、服务组件目标与 Docker 目标，相关数据 API 与代理必须拒绝访问；不得因本特性的可编辑配置下发、写回或密钥编辑流程而放行未登记目标。

### 需求 8：错误处理与可用性

**用户故事：** 作为用户，我希望保存失败时知道原因且原配置仍可用，以便放心修改。

#### 验收标准

1. 当保存失败（含服务端校验失败、磁盘写回失败，或客户端收到明确失败响应）时，在用户退出配置编辑器后，仪表盘主视图必须仍按写回前配置展示服务、书签、信息组件与设置，展示内容须与写回前磁盘上的配置五文件经配置加载器加载的结果一致，且不得将本次未成功保存的编辑内容应用到主视图；本条以写回前配置可被配置加载器成功加载为前提。
2. 写回接口与可编辑配置读取接口返回并用于界面展示的错误提示必须为简体中文；上述错误提示及配置编辑器据此展示的文案均不得包含完整明文密钥（密码、API key、token 等密钥字段的完整值）。
3. 当写回或可编辑配置读取因网络失败中断，或在客户端发起请求后 30 秒内未收到完整响应而被判定为超时时，配置编辑器必须展示简体中文失败提示，保留用户当前已编辑的全部字段内容，并在该次请求结束后允许用户再次触发保存；不得清空表单，也不得要求用户重新输入未修改的密钥字段。
4. 对任意写回请求（含合法与非法的可编辑配置载荷），系统不得因本特性处理路径上的未捕获异常导致进程退出；失败必须转化为结构化 API 错误响应（至少包含简体中文原因，可定位时还须包含字段路径或所属配置类别），且不得返回或展示保存成功状态。
5. 当用户在保存失败后关闭或确认错误提示时，配置编辑器必须继续保留失败前的已编辑内容并保持可编辑，不得自动退出编辑器，也不得丢弃未保存修改。
6. 当可编辑配置读取失败时，配置编辑器必须展示简体中文错误提示，仪表盘不得因此出现整页白屏；用户必须仍能退出配置编辑器；在写回前配置合法的前提下，退出后主视图必须仍可按写回前配置正常展示。

### 需求 9：往返与兼容

**用户故事：** 作为维护者，我希望前端保存与 YAML 手改可以共存，以便过渡期仍能用文件做备份和迁移。

#### 验收标准

1. 当服务进程未运行时，系统必须允许维护者使用外部文本编辑工具直接修改配置目录下的配置五文件，且不得要求必须通过写回接口才能变更磁盘配置。当配置五文件为合法 YAML、字段均落在第一阶段受支持子集、且密钥字段满足现有密钥占位或字面量存储规则时，无论该内容来自写回接口的成功持久化，还是来自服务停止期间的手工编辑，配置加载器均必须能通过与现有 `loadConfig()` 相同的读盘、解析、规范化与 AllowList 重建路径成功加载，并产出规范化配置与当次 AllowList。
2. 当配置目录中已存在仅使用第一阶段受支持字段子集、且密钥字段均符合现有占位或字面量规则的配置五文件时，系统必须允许用户在不经任何手工改写这些文件的前提下打开配置编辑器、获得可编辑配置，并在不修改任一字段（密钥字段以“保持原值”提交）的情况下成功完成一次保存；该次保存必须返回成功状态，且不得要求用户重新输入已有密钥。
3. 当用户对第 2 条所述既有配置在不改变各条目身份字段（纳入既有稳定 ID 计算的字段）且密钥字段均为“保持原值”的前提下成功保存一轮后，系统必须保证：保存前后基于磁盘配置五文件各执行一次 `loadConfig()` 时，对应条目的 `serviceId`、`probeId`、`widgetId`、`infoId` 保持不变；仪表盘主视图中的服务、书签与信息组件在受支持子集上的集合与顺序与保存前一致；后续数据 API 仍仅按该次磁盘配置经 `loadConfig()` 重建的 AllowList 鉴权与返回。
4. 当可编辑配置经写回接口成功序列化为配置五文件后，再经配置加载器加载得到规范化配置，并再次生成可编辑配置（密钥字段以密钥掩码或“已配置/未配置”表示）时，若用户不修改任何密钥字段并以“保持原值”再次保存成功，则磁盘上各密钥字段的存储值必须与该轮再次保存前完全一致（含原密钥占位文本或原字面量），不得被掩码字符串覆盖、清空或替换为其他值。
5. 当写回接口成功持久化时，系统必须使每个被写入的配置五文件均以 UTF-8 编码的纯文本 YAML 形式存在于配置目录中：文件必须可被常规文本编辑器打开与手工编辑，必须能被现有配置加载器的 YAML 解析路径成功解析，且不得以二进制封装、加密容器或其他非纯文本形式写入。
