# 实现计划：实时配置编辑

## 概述

在现有 monorepo 内增量实现「结构化配置编辑 + 实时写回生效」。实现语言为 **TypeScript**。持久化仍仅使用配置五文件；可编辑读取同时使用原始解析树与完整 `loadConfig()`；保存以原始解析树为基底保留未知键，并通过候选预验证、旧快照、持锁原子替换、失败回滚、正式目录复验与故障闸门保证一致性。前端保持 React、Vite、Tailwind CSS、CSS 主题令牌及既有 shadcn/ui 约束，不提供 YAML 编辑面。

本计划仅包含实现任务及现有 TypeScript 类型检查与构建检查点，不新增任何单元测试、属性测试、集成测试或测试工具任务。

## 任务

- [x] 1. 扩展领域契约与 API 路由常量
- [x] 1.1 在 `packages/domain` 定义 `EditableConfig` 与密钥字段 schema
    - 定义并导出 GET 使用的 `SecretFieldView`、PUT 使用的 `SecretFieldWrite`（`keep` / `set` / `clear`），以及设置、服务、书签、信息组件、Docker 端点和完整可编辑配置的 Zod schema 与推导类型
    - customapi `headers` 值使用相同密钥三态；写回契约不接收可用于旁路鉴权的 AllowList 或已解析密钥
    - Docker 端点契约仅覆盖增删改，不定义调序操作
    - _需求：2.2、2.3、3.3、3.5、7.1、7.4_
- [x] 1.2 扩展 API 路由常量与响应契约
    - 增加 `GET /api/config/editable` 与 `PUT /api/config` 契约，保持既有 `GET /api/config` 不变，并从包入口导出
    - 复用 `ErrorEnvelope` / `PublicError`，定义可编辑读取成功响应和不含敏感回显的 `{ ok: true }` 写回响应
    - _需求：2.2、4.1、4.9、8.2_

- [x] 2. 实现可编辑配置构建（`packages/config`）
- [x] 2.1 实现 `buildEditableConfig` / `getEditableConfig`
    - 同一请求读取 `readAndParseConfigSources()` 原始解析树并执行完整 `loadConfig()`；两者均成功后才构建可编辑结果，受支持字段语义以完整加载结果核对，密钥状态只取自原始树，不得从安全视图反推密钥
    - 覆盖 settings、服务分组/项、书签分组/项、信息组件和 Docker 端点；单文件或五文件缺失时遵循现有加载器空值/默认语义
    - 密钥及 customapi header 值仅返回已配置/未配置状态，不返回存储全文、占位全文或插值明文
    - 检测到含 URI `userinfo` 的 Docker 连接串时，以固定简体中文结构化错误拒绝；不得下发或记录原连接串、用户名或凭据
    - YAML、顶层结构或完整加载失败时返回脱敏中文错误；导出构建 API
    - _需求：1.2、1.5、2.1–2.4、2.6、7.1、7.2、8.2_

- [x] 3. 实现基于原始解析树的安全写回管线（`packages/config`）
- [x] 3.1 实现写回载荷解析与服务端校验
    - 校验完整结构化载荷及受支持字段约束，显式忽略或剔除 `allowList`、`secrets`、`resolved*` 等旁路字段
    - 拒绝含 `userinfo` 的 Docker 连接串，并使用不回显原串的固定中文字段错误
    - _需求：4.2、4.7、7.4、8.4_
- [x] 3.2 实现“基于原始解析树保留合并”
    - 以写回前原始解析树为基底，仅更新编辑器支持字段；未知或未支持键及其所属结构原样保留，不得以受支持子集整文件覆盖
    - 密钥和 customapi header 值统一处理：`keep` 原样保留磁盘字符串，`set` 写入新值，`clear` 或空字符串按空值语义清除
    - 列表条目按既有身份字段安全对应；存在别名、类型冲突、重复语义来源或无法无歧义对应时，在正式目录零修改时拒绝，并返回脱敏的简体中文字段路径
    - 保留身份字段与列表顺序，不添加默认 ID、编辑器元数据、排序键或其他影响稳定 ID 的隐式字段；无改动保存与全 `keep` 必须保持受支持语义、密钥原串和稳定 ID
    - _需求：2.5、4.2、4.5、4.6、4.10、7.2、7.3、9.1–9.4_
- [x] 3.3 实现五文件 UTF-8 YAML 序列化
    - 从保留合并后的五棵解析树生成加载器可识别的五文件，保持服务、书签、信息组件的分组与条目顺序，以及未修改 Docker 端点的相对顺序
    - 不引入影响 `serviceId`、`probeId`、`widgetId`、`infoId` 的隐式字段；无法安全序列化时拒绝，不静默丢弃未知键
    - _需求：3.4、4.3、9.3、9.5_
- [x] 3.4 实现候选预验证、快照、原子替换、回滚与临时资源清理
    - 在仅当前用户可访问的受限临时目录准备完整候选五文件，并对该目录执行完整 `loadConfig()`；失败时正式目录零修改
    - 写回前保存五文件的存在状态与完整旧快照；持锁逐文件以同目录临时文件和原子替换写入，并记录已替换文件
    - 任一替换失败或正式配置目录完整 `loadConfig()` 失败时，按旧快照回滚全部已替换文件；回滚后再次完整加载正式目录以验证旧配置
    - 成功响应前必须对正式配置目录再次执行完整 `loadConfig()` 并成功
    - 所有准备目录、替换临时文件及落盘快照资源均在最外层 `finally` 清理；错误与日志不得泄露路径、内容或敏感异常原文
    - _需求：4.2–4.4、4.8、4.9、4.11、5.1、5.5、8.1、8.2、8.4、9.5_
- [x] 3.5 实现写回互斥与 `CONFIG_FAULTED` 故障闸门
    - 写回锁覆盖原始树读取、旧快照、候选准备、替换、正式目录验证、必要回滚及故障状态更新，重叠写回串行完整执行
    - 回滚失败或回滚后旧配置验证失败时开启故障闸门；闸门开启后阻止后续写回，并供配置读取及数据 API 返回结构化错误，禁止使用混合配置建立安全视图或 AllowList
    - 仅在磁盘恢复且完整 `loadConfig()` 验证成功后解除闸门
    - _需求：1.6、5.5、6.1、6.4、6.5、8.1、8.4_
- [x] 3.6 组装完整写回入口函数
    - 串联载荷解析、加锁、原始树保留合并、候选完整加载、旧快照、五文件替换、正式目录完整加载、必要回滚和故障闸门
    - 仅在全部步骤成功后返回 `{ ok: true }`；任何失败均返回脱敏简体中文结构化错误且不声称成功
    - _需求：1.1、4.2–4.11、5.1、5.5、6.1、8.4_

- [x] 4. 检查点：配置层可编辑与写回
  - 对 `packages/domain`、`packages/config` 执行现有 TypeScript 类型检查和构建，确认无类型或构建错误；不新增或执行其他测试任务。
  - _需求：1.1–1.3、2.1–2.6、4.2–4.11、6.1、7.4、9.1–9.5_

- [x] 5. 实现服务端配置路由与每请求加载门禁
- [x] 5.1 扩展 `packages/server` 配置路由与公开输出保护
    - 增加 `GET /api/config/editable` 与 `PUT /api/config`，接入配置层 API、schema 断言、写回锁和故障闸门；未捕获异常转为结构化简体中文错误且不得导致进程退出
    - 统一约束入口 HTML、所有 API 正文与响应头、新增接口、错误和日志：不得泄露完整密钥、存储侧敏感值、原始请求、敏感异常原文、Docker `userinfo`、临时路径或临时内容
    - 日志仅记录稳定错误码、配置文件逻辑名和脱敏字段路径；写回响应不得回显 `set` 新值或请求片段
    - _需求：1.4、1.5、2.1–2.4、4.1、4.6–4.9、7.1–7.3、8.2、8.4_

    - 逐一核查配置读取、HTTP 探测、Docker、信息组件和服务组件；仅在需要时最小修正为每个请求执行完整 `loadConfig()` 并使用当次规范化配置与 AllowList，不得跨请求缓存或使用旁路真相
    - 保持未登记目标拒绝行为；故障闸门开启时，上述配置和数据 API 均返回结构化简体中文错误，不得使用混合磁盘配置鉴权或伪造成功数据
    - _需求：1.2、1.3、1.6、5.1、5.2、5.5、7.5_

    - 不在客户端日志或错误中记录原始请求、密钥、Docker 敏感连接串或服务端敏感异常

## 备注

- 本计划不包含任何新增自动化测试或测试工具任务；检查点仅执行现有 TypeScript 类型检查与构建。
- 保存以原始解析树为基底，仅更新支持字段；未知或未支持键必须原样保留，无法安全对应时必须在正式目录零修改时拒绝。
- 服务、书签和信息组件调序使用上移/下移，不引入拖拽库；Docker 端点仅增删改。
- 配置编辑器保持结构化非 YAML，并遵循 React、Vite、Tailwind CSS、CSS 主题令牌和既有 shadcn/ui 约束。
- 每个任务只实施其可追溯需求所需的最小改动。
