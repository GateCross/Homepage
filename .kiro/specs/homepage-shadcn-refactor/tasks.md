# 实现计划

## 概述

本计划在空的 `pnpm` 工作区中实现第一阶段仪表盘，采用 TypeScript，并保持 `packages/domain`、`packages/config`、`packages/adapters`、`packages/server` 与 `apps/web` 的逻辑边界。必做主链路为：工程骨架 → 共享契约、稳定 ID 与领域纯函数 → 配置加载与 AllowList → 核心检查点 → 适配器、信息提供者与 Docker → 服务端 API → 前端 API 客户端与应用壳 → 页面组件 → 搜索 → 全链路自动化测试 → 生产构建与启动冒烟 → 最终检查点。

所有普通勾选项均为第一阶段必做任务，包括全部自动化测试。设计中的正确性属性 1–20 必须逐项实现属性测试，每项至少执行 100 次生成用例；测试可以分批编写和运行，但最终必须全部通过。任何必做测试均不得跳过。

第一阶段不得加入配置热加载、YAML 写回、ICMP Ping、Docker stats、`immich`、`gitea`、`caddy`、认证、完整 i18n 或旧 UI 兼容。

- [x] 2. 实现共享契约、确定性稳定 ID 与领域纯函数

    - 使用 Zod 作为唯一运行时 schema 实现；TypeScript 类型必须由 schema 推导或与 schema 保持单一来源，禁止仅依赖编译期类型。
    - `ResourcesInfoResponse` 的资源项使用可区分联合：可用项含 `[0,100]` 的 `percent`，不可用磁盘项含 `status: "unavailable"` 与中文消息；每项保留可区分 CPU、内存或磁盘路径的稳定标识和标签。
    - _需求：3.3、3.4、6.4、7.10、8.2、9.2、9.3_
  - [x] 2.2 锁定各 API 的成功结构、错误结构与状态码
    - `/api/config` 成功返回通过 schema 校验的 `NormalizedConfig`；probe、docker、info 成功分别返回对应共享响应；widget 使用设计中的 `ServiceWidgetResult` 成功分支。
    - 所有路由错误统一返回 `{ ok: false, error: PublicError }`；配置 YAML 语法或结构错误返回 422，未登记 ID、端点或容器返回 403，外部服务失败返回 502，外部超时返回 504，未处理的服务端错误返回 500。
    - API 404 返回 JSON `PublicError`，不得进入 SPA fallback 或返回 HTML。
    - _需求：3.4、7.11、8.5、9.8、12.1_
  - [x] 2.3 定义统一确定性稳定 ID 算法
    - 为 `serviceId`、`probeId`、`widgetId`、`infoId` 实现统一的规范串构造器与稳定哈希函数；使用 SHA-256，并采用类型前缀加前 24 个十六进制字符作为公开 ID。
    - 规范串只使用配置来源类型、文件内分组路径或索引、条目索引、规范化类型和必要目标身份；不得使用随机数、时间戳、进程内自增状态、绝对磁盘目录或 worker 状态。
    - `serviceId` 使用 `services`、分组索引、服务索引与规范化有效 `href`；服务名称、描述、图标、`weight`、`target` 等展示字段变化不得改变 ID。
    - `probeId` 使用服务位置与规范化探测 URL；`expectedStatus`、超时和展示字段变化不改变 ID，探测 URL 或条目位置变化必须改变 ID。
    - `widgetId` 使用服务位置、被选中组件在有效组件列表中的索引、规范化组件类型和规范化目标 URL；mappings、headers、账号、密码、token、API key 及其变化均不改变 ID。
    - `infoId` 使用 `widgets` 条目索引、规范化信息类型和提供者目标身份；Open-Meteo 的坐标与时区、resources 的规范化磁盘路径集合属于目标身份，展示格式与标签不属于目标身份。
    - 密钥明文、qBittorrent `SID`、Emby token 与 Custom API header 值不得进入规范串、哈希输入、公开 ID 或调试输出。
    - _需求：3.3、3.9、7.11、9.10_
  - [x] 2.4 编写稳定 ID 单元、属性与跨加载集成测试
    - 验证同一规范化配置连续两次 `loadConfig()` 产生完全相同的四类 ID，并通过启动独立 Node 子进程或 worker 验证跨进程结果相同。
    - 验证数据 API 每次重建 AllowList 后，旧页面快照中的 `probeId`、`widgetId`、`infoId` 对未变化配置仍能命中。
    - 验证无关展示字段变化不破坏数据 API ID；目标 URL、条目位置或上文明确纳入身份的字段变化会得到新 ID。
    - 使用随机高熵密钥验证密钥值变化不改变 ID，且 ID、规范串和测试日志均不包含完整密钥。
    - _需求：3.6、3.9、7.11、9.10、13.6、13.7_
  - [x] 2.5 实现基础领域纯函数

    - 服务网格最小宽度为 240px，书签网格最小宽度为 160px；列数始终不小于 1。
    - _需求：4.3、4.6、5.2、5.4、10.2、10.3、11.1–11.3、11.7_
  - [x] 2.6 实现主题、探测、密钥和资源纯函数

    - 未配置 `expectedStatus` 时 100–599 均为可达；DNS、连接、TLS 和超时均为不可达。
    - _需求：2.1–2.6、3.7、6.4、7.5–7.10、8.2_
  - [x] 2.7 编写领域正确性属性测试
    - 分别实现属性 3、4、5、6、9、10、11、12、13、15、18、19、20；每项至少运行 100 次生成用例。

    - YAML 语法、顶层结构和受支持字段错误转换为中文 `PublicError`，至少包含文件名和原因，可定位时包含字段路径或行列。
    - 不实现文件监听、热加载、序列化或写回。
    - _需求：3.1–3.6_
  - [x] 3.2 实现设置与布局规范化

    - `language`、`headerStyle` 和未知字段读后安全忽略；标题无效时使用非空中文默认标题。
    - _需求：2.6、2.7、3.10、11.1_
  - [x] 3.3 实现服务基础规范化
    - 保留分组和条目声明顺序，解析 `name`、`href`、`target`、`icon`、`description`、`ping`、`showStats`、`weight` 等基础字段并生成 `serviceId`。

    - 应用稳定 `weight` 排序，同时保留 ID 所依据的原始配置位置，避免排序改变 ID。
    - _需求：4.1–4.9_
  - [x] 3.4 实现探测规范化和稳定 `probeId`
    - 支持绝对 URL 形式的 `siteMonitor` 与启用标志形式；启用标志仅在 `href` 有效时登记。

    - _需求：7.1–7.4_
  - [x] 3.5 实现 Docker 端点与容器登记
    - 解析 `docker.yaml` 中命名端点，只接受 `unix:///var/run/docker.sock` 形式的 Unix socket 和 `tcp://host:port` 形式的 TCP 端点。
    - 仅当服务同时声明有效 `server` 与非空 `container` 时登记 `${server}::${container}`；未声明端点不进入 AllowList。
    - `showStats` 不触发任何 stats 能力。
    - _需求：8.1、8.6、8.7_
  - [x] 3.6 实现服务组件规范化和稳定 `widgetId`

    - 严格执行密钥整值 `${ENV_VAR}` 插值；环境变量缺失或空时生成依赖该组件的中文配置错误。
    - 生成稳定 `widgetId`；目标 URL、服务端 secrets 和 options 仅进入 `AllowList.widgetTargets`，浏览器配置不得包含账号、密码、token、API key、headers 或插值结果。
    - Custom API 首期锁定仅 `GET`：缺省为 `GET`，任何其他 method 产生局部中文配置错误；不接受或生成请求体。

  - [x] 3.7 实现信息组件规范化和稳定 `infoId`
    - 仅支持 `datetime`、`openmeteo`、`resources`；其他类型标记暂不支持且不登记数据目标。
    - 为每项生成稳定 `infoId`；Open-Meteo 坐标和时区、resources 磁盘路径等只存入当次 `AllowList.infoTargets`，前端获得必要的非敏感展示配置。
    - 规范化 datetime 的 IANA 时区和受支持格式；非法格式安全回退且不抛未捕获异常。
    - _需求：6.1–6.7_

    - 单条名称或链接无效时在原区域生成局部中文错误，不使整份文件或其他分组失败。
    - _需求：5.1–5.6_
  - [x] 3.9 汇总安全视图和当次 AllowList

    - 数据 API 每次调用必须重新执行 `loadConfig()` 并使用当次 AllowList 鉴权，不维护跨请求的全局授权真相。
    - 在返回前执行运行时 schema 校验和密钥扫描；校验失败转为不含内部数据和密钥的中文服务端错误。

    - 覆盖五文件缺失、单文件缺失、局部条目错误、探测超时边界、稳定排序、组件选取、整值插值失败、未知类型和 Docker 登记。

  - 运行领域、共享 schema、稳定 ID、配置和 AllowList 测试；属性 1–13 中已经进入主链路的每项必须达到至少 100 次并全部通过。

  - 检查命令任一失败时任务失败；不得以手工确认替代或继续绕过。
  - _需求：3.1–3.10、7.11、8.6、9.10、13.1–13.9_
