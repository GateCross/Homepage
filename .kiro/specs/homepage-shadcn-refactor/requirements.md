# 需求文档

## 概述

本功能基于当前 Homepage 仓库，使用 React、Tailwind CSS 与 shadcn/ui 全量重建自托管仪表盘界面。实施方式不是逐个替换旧 UI，也不延续旧组件树和旧 CSS，而是建立新的精简页面与组件体系；仅复用确有价值的配置加载、服务端代理、Docker 查询与图标处理等底层能力。复用底层能力不得扩大第一阶段范围或恢复已推迟功能。

参考实例 `http://192.168.50.10:9999/` 仅作为功能与内容范围锚点，用于形成第一阶段验收配置样例；不得参考或复制参考实例及旧 Homepage 的视觉风格、组件结构或 CSS。第一阶段只支持当前实际使用的 YAML 配置子集和明确列出的组件类型，不追求全部旧配置边界兼容，也不以保持与上游 Homepage 无冲突合并或持续升级能力为目标。

第一阶段交付范围包括服务与书签展示、三态主题、日期时间、Open-Meteo 天气、主机资源、HTTP 在线探测、Docker 容器运行状态、限定类型的服务组件、快捷搜索、响应式布局、中文状态反馈，以及可在局域网通过 HTTP 访问的部署产物。

## 术语表

| 术语 | 说明 |
| --- | --- |
| 仪表盘 | 本功能重建后对外呈现的单页应用 |
| 配置加载器 | 读取、解析、校验并规范化受支持 YAML 配置子集的服务端模块 |
| 规范化目标 | 配置加载器根据 `siteMonitor` 语义解析并校验后，登记为允许探测对象的绝对 HTTP 或 HTTPS URL |
| 服务分组 | 一组有序的服务卡片 |
| 服务卡片 | 展示服务名称、链接、图标、描述、状态与可选指标的界面单元 |
| 书签分组 | 一组有序的书签条目 |
| 书签条目 | 展示名称、链接、图标或缩写及可选描述的界面单元 |
| 信息组件 | 展示日期时间、天气或主机资源信息的组件 |
| 服务组件 | 挂载在服务卡片上并展示外部服务指标的组件 |
| 服务组件适配器 | 在服务端调用外部 API，并将不同响应结构转换为统一指标模型的模块 |
| 统一指标模型 | 至少包含稳定标识、中文标签、值、可选单位和可选状态的服务指标表达 |
| HTTP 探测 | 对规范化目标发起请求，并区分可达性与可选业务状态的能力 |
| Docker 状态 | 通过已声明 Docker 端点查询容器运行状态的能力，不包含第一阶段推迟的 CPU 与内存实时统计 |
| 代理接口 | 仅面向已声明目标执行服务端请求，并向浏览器返回安全结果的接口 |
| 主题模式 | `system`、`light`、`dark` 三种主题选择；默认值为 `system` |
| 容器容量 | 当前网格容器按照规定的条目最小宽度能够容纳的列数 |
| 验收配置样例 | 从参考实例提取并用于验收的配置数据；其中坐标、路径、端点名、分组名和列数均不是程序常量 |

## 需求

### 需求 1：重建范围与技术基础

**用户故事：** 作为维护者，希望建立独立、精简的现代化界面体系，以便降低旧 UI 结构对后续迭代的限制。

#### 验收标准

1. 仪表盘必须采用 React、Tailwind CSS、CSS 主题令牌和 shadcn/ui 构建全新界面。
2. 新界面不得导入旧 UI 组件、继承旧组件树或加载旧 UI CSS，也不得以逐个替换旧 UI 作为实施方式。
3. 仪表盘可以复用当前仓库中确有价值的配置加载、服务端代理、Docker 查询和图标处理能力，但新页面与业务组件必须形成独立、精简的组织结构；底层能力复用不得恢复实时热加载、ICMP Ping、Docker CPU 或内存统计、YAML 写回、推迟支持的服务组件或旧 UI 兼容能力。
4. 通用按钮、对话框、输入框、浮层、加载占位等交互必须优先采用适合的 shadcn/ui 基元。
5. 当通用 shadcn/ui 基元不适合业务表达时，业务专用组件必须使用 Tailwind CSS 与主题令牌实现，不得为了使用组件而强制套用不合适的 Card、Badge 或 Progress。
6. 仪表盘不得复制或模仿旧 Homepage 与参考实例的视觉风格、组件结构或 CSS。
7. 当生产构建完成时，系统必须生成无需开发服务器即可启动的 Node 服务或可部署的容器镜像。
8. 当生产产物启动并绑定局域网可访问地址时，同一局域网内的客户端必须能够通过 HTTP 获取仪表盘入口页面，且入口页面引用的静态资源必须能够成功加载。

### 需求 2：主题与基础页面体验

**用户故事：** 作为用户，希望界面适应系统主题并记住手动选择，以便在不同环境下获得一致、舒适的浏览体验。

#### 验收标准

1. 当用户首次打开仪表盘且不存在有效的已保存主题值时，主题模式必须回退为 `system`，活动主题必须与操作系统当前色彩方案一致；空值及 `system`、`light`、`dark` 之外的值均为无效主题值。
2. 在主题模式为 `system` 的过程中，当操作系统色彩方案发生变化时，活动主题必须在 1 秒内同步变化且无需刷新页面。
3. 当用户选择 `light` 或 `dark` 时，仪表盘必须在 1 秒内应用对应主题、持久化该选择并停止跟随操作系统色彩方案。
4. 当用户刷新页面时，仪表盘必须恢复有效的已持久化手动主题模式；如果已保存值无效，则必须回退为 `system`。
5. 当用户切换回 `system` 时，仪表盘必须在 1 秒内恢复对操作系统当前及后续色彩方案变化的跟随。
6. 当 `settings.yaml` 提供去除首尾空白后非空的 `title` 时，仪表盘必须将浏览器文档标题设置为去除首尾空白后的值；如果未提供有效 `title`，则必须使用非空的中文默认标题。
7. 在配置了背景图的情况下，仪表盘必须将背景图显示在主内容后方；如果背景图加载失败，则必须回退到当前主题的默认背景，并保持正文与控件可见、可读和可操作。
8. 仪表盘的通用错误、空状态、加载状态、主题控件与搜索控件必须使用简体中文文案。

### 需求 3：配置读取、校验与安全

**用户故事：** 作为部署者，希望继续使用当前实际配置文件的受支持子集，以便以可控成本迁移现有内容。

#### 验收标准

1. 配置加载器必须只读取和处理 `settings.yaml`、`services.yaml`、`bookmarks.yaml`、`widgets.yaml`、`docker.yaml` 中第一阶段实际使用的字段与结构子集。
2. 当任一受支持配置文件不存在时，配置加载器必须将对应类别规范化为空配置或内置默认配置，并继续处理其他文件。
3. 当合法配置被加载时，配置加载器必须生成供页面和代理接口使用的规范化配置，且不得要求将配置写回磁盘。
4. 当配置包含 YAML 语法错误、无效结构或受支持字段的无效值时，配置加载器必须返回结构化、可读的中文错误；错误必须至少包含文件名和原因，在能够定位时还必须包含字段路径或 YAML 行列位置。
5. 对任意合法的受支持配置，配置加载与页面渲染不得产生未捕获异常；对任意非法配置，页面必须展示对应错误且不得出现整页白屏。
6. 当用户修改 YAML 后刷新浏览器时，仪表盘必须在该次刷新过程中重新读取当前磁盘配置并反映修改结果；刷新发生前不得自动加载磁盘变更。
7. 当密钥字段的整个值严格符合 `${ENV_VAR}` 占位形式时，配置加载器必须仅在服务端使用对应环境变量值替换该占位；嵌入其他文本的占位内容不得在第一阶段执行插值。
8. 如果密钥占位所引用的环境变量不存在或为空，则依赖该密钥的功能必须返回中文配置错误。
9. 浏览器收到的入口页面、配置数据、响应头、响应正文和错误信息不得包含配置加载器已加载的完整明文密钥、密码、令牌或 API 密钥值。
10. 当配置中出现第一阶段未支持的字段或类型时，配置加载器必须安全忽略对应内容，或将对应组件标记为“暂不支持”，并继续处理其他合法内容；系统无需为每个未知字段都显示可见提示。

### 需求 4：服务分组与服务卡片

**用户故事：** 作为用户，希望按配置查看有序的服务入口，以便快速访问局域网服务与应用。

#### 验收标准

1. 当有效 `services.yaml` 被加载时，配置加载器必须保留服务分组的声明顺序，并解析每组中的服务卡片。
2. 每张服务卡片的名称去除首尾空白后必须非空，页面必须展示去除首尾空白后的名称，并在配置存在时展示图标和描述。
3. 服务卡片的 `href` 在第一阶段仅当其为绝对 HTTP 或 HTTPS URL 时有效。
4. 当服务卡片具有有效 `href` 时，用户点击卡片必须按 `target` 指定方式打开链接，未配置 `target` 时默认在新标签页打开。
5. 当服务卡片没有有效 `href` 时，用户点击卡片不得触发页面导航。
6. 当同一服务分组中存在数值型 `weight` 时，服务卡片必须按 `weight` 降序进行稳定排序；相同 `weight` 的卡片必须保持原声明顺序，未配置 `weight` 的卡片必须保持相对顺序并排在已配置 `weight` 的卡片之后。
7. 配置加载器必须接受第一阶段所需的 `name`、`href`、`target`、`icon`、`description`、`siteMonitor`、`expectedStatus`、`ping`、`server`、`container`、`showStats`、`weight`、`widget` 和 `widgets` 字段。
8. 当配置中存在 `ping` 时，仪表盘必须安全忽略该探测配置或显示“暂不支持”；`ping` 不得使服务卡片进入“不可达”状态，也不得影响其他卡片。
9. 当配置中存在 `showStats: true` 时，配置加载器必须接受该配置，但仪表盘不得展示任何 Docker CPU、内存或其他资源统计。

### 需求 5：书签分组与书签条目

**用户故事：** 作为用户，希望按分组查看常用书签，以便快速打开目标站点。

#### 验收标准

1. 当有效 `bookmarks.yaml` 被加载时，配置加载器必须保留书签分组及条目的声明顺序。
2. 每个书签条目的名称去除首尾空白后必须非空，`href` 必须为绝对 HTTP 或 HTTPS URL。
3. 书签条目必须展示去除首尾空白后的名称，并在配置存在时展示描述。
4. 当书签配置了可正常加载和渲染的 `icon` 时，书签条目必须展示对应图标；未配置 `icon` 或图标加载、渲染失败时，必须回退到非空 `abbr`，不存在非空 `abbr` 时必须回退到名称的首个字符。
5. 当用户点击书签条目时，仪表盘必须按 `target` 指定方式打开 `href`，未配置 `target` 时默认在新标签页打开。
6. 当单个书签名称或 `href` 无效时，仪表盘必须在该书签原属区域显示局部中文配置错误，并继续展示该区域及其他区域中可正常解析的条目。

### 需求 6：信息组件

**用户故事：** 作为用户，希望在首页查看时间、天气和主机资源，以便快速了解当前环境。

#### 验收标准

1. 第一阶段信息组件类型白名单必须为且仅为 `datetime`、`openmeteo`、`resources`。
2. 当配置包含 `datetime` 时，信息组件必须按照配置的时区以及受支持的日期和时间格式展示当前时间，显示值与该时区当前时间的偏差不得超过 60 秒。
3. 当配置包含 `openmeteo` 时，信息组件必须通过 Open-Meteo 展示配置坐标和时区对应的天气摘要，至少包括摄氏温度和天气状况文字或图标。
4. 当配置包含 `resources` 时，信息组件必须展示 CPU、内存以及配置中各磁盘路径的使用率；每个显示值必须明确对应 CPU、内存或具体磁盘路径，百分比必须处于 0 至 100 范围。
5. 当天气或资源数据正在请求时，对应信息组件必须显示中文加载状态。
6. 当天气或资源数据请求失败时，错误必须局限于对应信息组件；对应信息组件必须显示中文错误状态，其他信息组件、服务和书签必须继续渲染并可交互。
7. 当配置包含白名单之外的信息组件类型时，仪表盘必须显示“暂不支持”或安全忽略该组件，并继续渲染其余内容。

### 需求 7：HTTP 在线探测

**用户故事：** 作为用户，希望准确区分服务的可达性与业务状态，以便避免将需要认证或返回错误页的服务误判为不可达。

#### 验收标准

1. 当服务卡片的 `siteMonitor` 为绝对 HTTP 或 HTTPS URL 字符串时，配置加载器必须将该 URL 规范化并登记为探测目标，不要求服务卡片同时具有 `href`。
2. 当服务卡片的 `siteMonitor` 为启用标志时，配置加载器必须将该服务卡片的有效 `href` 规范化并登记为探测目标，不要求另行提供 `siteMonitor` URL。
3. 当服务卡片具有可探测的规范化目标时，代理接口必须只对该规范化目标发起 HTTP 探测；目标授权校验必须基于规范化后的允许目标，而不是要求请求目标与 `siteMonitor` 原始值完全一致。
4. HTTP 探测的默认超时时间必须为 10 秒；如果受支持配置提供有效的超时覆盖值，则 HTTP 探测必须使用该配置值。
5. 当探测在有效超时时间内收到状态码为 100 至 599 的 HTTP 响应时，HTTP 探测必须将目标判定为可达；401、403、404 及 5xx 响应不得在默认规则下被判定为不可达。
6. 当探测发生 DNS 解析失败、连接失败、TLS 握手失败或超时时，HTTP 探测必须将目标判定为不可达。
7. 在未配置 `expectedStatus` 的情况下，状态码为 100 至 599 的 HTTP 响应必须映射为“可达”。
8. `expectedStatus` 必须同时支持单个状态码和包含边界的状态码范围。
9. 在配置了 `expectedStatus` 的情况下，匹配单个状态码或状态码范围的有效 HTTP 响应必须映射为“可达”，不匹配预期的有效 HTTP 响应必须映射为“可达但状态异常”。
10. 服务卡片的 HTTP 探测状态必须为“加载中”“可达”“可达但状态异常”“不可达”四种互斥状态之一。
11. 代理接口必须拒绝任何未登记在当前规范化配置中的探测目标，并且不得对被拒绝目标发出网络请求。
12. 当单个探测请求失败时，仅对应服务卡片的状态必须受到影响，其他内容必须继续展示与交互。

### 需求 8：Docker 容器状态

**用户故事：** 作为用户，希望查看容器是否运行，以便识别已停止或暂时不可查询的容器服务。

#### 验收标准

1. 当服务卡片同时配置 `server` 与 `container` 时，代理接口必须根据 `docker.yaml` 中同名端点查询指定容器状态。
2. Docker 状态必须为“加载中”“运行中”“已停止”“不可用”四种互斥状态之一。
3. 当 Docker 返回的容器状态明确表示正常运行时，对应服务卡片必须显示“运行中”。
4. 当 Docker 返回的容器状态未明确表示正常运行时，对应服务卡片必须显示“已停止”。
5. 当端点未声明、端点不可达、容器不存在、查询结果不确定或查询失败时，对应服务卡片必须显示“不可用”，其他页面内容必须继续展示与交互。
6. Docker 查询代理必须只允许访问 `docker.yaml` 中已声明的命名端点和服务配置中已声明的容器。
7. 第一阶段不得展示容器 CPU、内存或其他资源统计，即使现有服务配置包含 `showStats: true`。

### 需求 9：服务组件与适配器

**用户故事：** 作为用户，希望在部分服务卡片上查看统一呈现的关键指标，以便无需打开外部应用即可了解摘要状态。

#### 验收标准

1. 第一阶段服务组件类型白名单必须为且仅为 `qbittorrent`、`emby`、`customapi`。
2. 服务组件适配器必须在服务端将不同外部 API 响应转换为统一指标模型，前端不得直接依赖第三方 API 的原始响应结构。
3. 同一个服务组件内的每个统一指标标识必须唯一；统一指标模型必须能够表达稳定标识、中文标签、值、可选单位和可选状态。
4. 当配置 `qbittorrent` 时，服务组件必须至少展示下载速率和上传速率，并对两种速率使用一致的单位体系。
5. 当配置 `emby` 时，服务组件必须展示正在播放会话总数以及最多 5 条正在播放会话摘要；没有正在播放会话时必须显示中文空状态。
6. Emby 服务组件第一阶段不得展示历史播放记录或媒体库统计。
7. 当配置 `customapi` 时，服务组件适配器必须按照 `mappings` 将响应字段转换为统一指标；单个映射缺失、取值失败或格式化失败时，仅该指标必须显示不可用状态或被省略，整个组件及其他映射不得进入失败状态。
8. 当服务组件请求正在进行时，对应卡片必须显示中文加载状态；当请求失败或超时时，仅对应卡片必须显示中文错误状态。
9. 当配置 `immich`、`gitea`、`caddy` 或其他非白名单服务组件时，仪表盘必须显示“暂不支持”或安全忽略对应组件，并保留服务卡片的基础展示与链接跳转；代理接口不得为非白名单类型发出指标请求。
10. 服务组件必须仅通过代理接口访问外部服务，代理接口必须仅允许访问 YAML 中相应服务组件已声明的目标。
11. `qbittorrent`、`emby`、`customapi` 三种服务组件必须分别独立注册；信息组件也必须能够按 `type` 独立注册，新增类型不得要求修改无关类型的业务实现。

### 需求 10：快捷搜索

**用户故事：** 作为用户，希望通过键盘或页面入口搜索服务与书签，以便快速定位目标。

#### 验收标准

1. 当用户按下 Windows 或 Linux 的 `Ctrl + K`、macOS 的 `Meta + K`，或点击搜索入口时，搜索浮层必须打开并将焦点置于输入框。
2. 当搜索词为空时，搜索浮层必须展示当前已加载的全部服务和书签，每个服务或书签最多出现一次。
3. 当搜索词非空时，搜索必须针对页面可见名称执行不区分大小写的连续子串匹配；搜索结果必须完整包含且仅包含命中项，每个服务或书签最多出现一次。
4. 当没有匹配结果时，搜索浮层必须显示中文空状态并保持打开。
5. 当用户选择具有有效 `href` 的结果时，仪表盘必须按照对应条目的 `target` 规则打开链接并关闭搜索浮层。
6. 当用户选择没有有效 `href` 的结果时，仪表盘不得导航，搜索浮层必须保持打开并显示中文无效链接提示。
7. 当用户按下 `Escape` 时，搜索浮层必须关闭且不得改变当前页面地址。

### 需求 11：响应式布局

**用户故事：** 作为用户，希望服务和书签网格根据可用空间自适应，以便在桌面、平板和手机上保持内容可读。

#### 验收标准

1. 配置中的 `layout.<GroupName>.columns` 必须表示对应分组的最大列数，不得被解释为任何宽度下的强制列数。
2. 每个分组的实际列数必须取配置最大列数与当前容器容量中的较小值，并且始终不得小于 1；空集合由对应区域的空状态处理，布局列数仍不得小于 1。
3. 服务卡片的布局计算必须采用 240px 的最小宽度，书签条目的布局计算必须采用 160px 的最小宽度。
4. 当容器宽度不足以容纳配置最大列数时，网格必须自动减少列数并保持条目内容可读。
5. 在任意受支持视口宽度下，网格项不得相互重叠或被网格容器裁切，网格布局不得导致页面产生水平滚动。
6. 当 `useEqualHeights` 为 `true` 时，同一网格行中服务卡片的外部高度必须一致；当 `useEqualHeights` 未启用时，每张服务卡片的高度必须由自身内容决定。
7. 对任意有效配置最大列数和任意正数容器容量，实际列数必须始终大于或等于 1，且不得超过配置最大列数或容器容量。

### 需求 12：局部状态与故障隔离

**用户故事：** 作为用户，希望单个数据源或配置项失败时仍能使用其余首页内容，以便局部故障不会中断整个仪表盘。

#### 验收标准

1. 每个异步信息组件、服务组件、HTTP 探测和 Docker 查询必须区分并展示适用的中文加载、空和错误状态；首次请求仍在等待结果时必须显示加载状态，不得显示空状态或错误状态。
2. 当单个组件、探测、容器查询或页面区域失败时，仪表盘必须将错误限制在对应区域，不得清空其他区域或阻止其他区域继续加载、渲染和交互。
3. 当配置类型未受支持时，仪表盘必须显示中文“暂不支持”提示或安全忽略对应内容，且不得产生未捕获异常。
4. 当服务或书签集合为空时，对应区域必须显示中文空状态，而不是无说明的空白区域。
5. 当失败的数据请求在后续重新请求时成功，对应区域必须退出错误状态并恢复正常数据展示。

### 需求 13：正确性与安全测试约束

**用户故事：** 作为维护者，希望关键规则可通过自动化方式验证，以便重构后持续发现配置、状态和安全边界问题。

#### 验收标准

1. 自动化测试必须验证：任意合法的受支持配置不会触发未捕获异常，任意非法配置会产生至少包含文件名和原因的结构化中文错误，并在能够定位时包含字段路径或 YAML 行列位置。
2. 自动化测试必须验证：对任意服务卡片序列，按 `weight` 排序后结果满足降序规则，并保持相同权重及未配置权重条目的稳定相对顺序。
3. 自动化测试必须验证：对任意服务、书签集合和搜索词，结果按照可见名称进行不区分大小写的连续子串匹配，完整包含且仅包含命中项，并且不包含重复结果。
4. 自动化测试必须验证：对任意有效配置最大列数和正数容器容量，包括条目集合为空的情况，实际列数始终不小于 1，且不超过两者中的较小值。
5. 自动化测试必须验证：任意主题状态转换均满足无效保存值回退 `system`、`system` 在 1 秒内跟随、手动模式在 1 秒内应用并停止跟随、刷新后恢复有效手动模式以及切回 `system` 后恢复跟随的规则。
6. 自动化测试必须验证：代理接口对任意未声明探测目标、外部服务目标、Docker 端点或容器请求均返回拒绝结果，并且不对被拒绝目标发出网络请求。
7. 自动化测试必须验证：任何发送给浏览器的入口页面、配置数据、错误信息、响应头和响应正文均不包含配置加载器已经加载的完整明文密钥值。
8. 自动化测试必须覆盖 HTTP 探测分类：状态码 100 至 599 在未配置 `expectedStatus` 时均为“可达”；`expectedStatus` 单个状态码与状态码范围分别具有匹配为“可达”和不匹配为“可达但状态异常”的结果；DNS 解析失败、连接失败、TLS 握手失败和超时均为“不可达”。
9. 自动化测试必须验证：HTTP 探测在任一时刻只处于“加载中”“可达”“可达但状态异常”“不可达”中的一种状态，Docker 查询在任一时刻只处于“加载中”“运行中”“已停止”“不可用”中的一种状态。

## 非目标（第一阶段明确不做）

- `services.yaml` 或 `bookmarks.yaml` 的序列化、格式化、写回及往返等价保证
- 配置文件实时热加载、文件监听与自动刷新
- ICMP Ping 探测；配置中的 `ping` 仅安全忽略或显示“暂不支持”
- `immich`、`gitea`、`caddy` 服务组件
- Docker 容器 CPU、内存及其他资源统计；`showStats: true` 可保留但不触发统计展示
- 白名单之外的上游服务组件与信息组件
- Docker 标签自动发现、Kubernetes 集成与 Proxmox 集成
- 完整多语言支持、内置 OIDC 或密码认证、多用户能力、自定义 CSS 或 JavaScript 注入编辑器
- 所有旧 Homepage 配置边界的兼容保证
- 与上游 Homepage 持续无冲突合并或升级的保证
- 旧 Homepage 或参考实例视觉、组件树、CSS 与布局美学的复制或兼容
- 移动端原生应用
